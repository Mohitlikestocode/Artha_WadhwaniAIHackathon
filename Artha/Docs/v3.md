## SECTION A: THE UNIVERSAL JARGON TOOLTIP SYSTEM

> Implement this across EVERY screen in KarmaLoop. No exceptions.
> This is the single feature that makes KarmaLoop feel like it actually cares about
> first-generation learners.

### The Problem It Solves

A first-gen graduate from Nagpur encounters the phrase "sync with the stakeholders
on the RACI before the sprint retrospective kicks off." She doesn't know:
- What "sync" means in a professional context (it's not just calendar sync)
- What a "stakeholder" is beyond the MBA textbook definition
- What a RACI is
- What a "sprint retrospective" is
- Why "kicks off" is used this way

On every other platform, she either guesses, feels shame, and disengages — or she
opens a second browser tab to Google, breaking her immersion completely.

KarmaLoop removes both options. She simply hovers over (or long-presses on mobile)
any underlined term and gets an instant, contextual, jargon-free explanation.

### Technical Implementation

**Step 1 — Term Detection Pipeline:**
Every piece of text rendered in KarmaLoop (character messages, documents, dashboards,
emails, scenario descriptions) passes through a pre-render processing step.

```javascript
// jargonProcessor.js
import { JARGON_LIBRARY } from './jargonLibrary';

export function processTextForJargon(rawText) {
  let processed = rawText;
  
  // Sort by length descending to match multi-word terms first
  const sortedTerms = Object.keys(JARGON_LIBRARY).sort((a, b) => b.length - a.length);
  
  sortedTerms.forEach(term => {
    const regex = new RegExp(`\\b(${escapeRegex(term)})\\b`, 'gi');
    processed = processed.replace(regex, (match) => {
      const entry = JARGON_LIBRARY[term.toLowerCase()];
      return `<jargon 
        term="${match}" 
        definition="${entry.definition}" 
        example="${entry.example}"
        category="${entry.category}"
      >${match}</jargon>`;
    });
  });
  
  return processed;
}
```

**Step 2 — JargonTooltip React Component:**

```jsx
// JargonTooltip.jsx
import { useState, useRef } from 'react';
import { motion, AnimatePresence } from 'framer-motion';

export function JargonTooltip({ term, definition, example, category, children }) {
  const [visible, setVisible] = useState(false);
  const [position, setPosition] = useState({ x: 0, y: 0 });
  const ref = useRef(null);

  const handleInteraction = (e) => {
    const rect = ref.current.getBoundingClientRect();
    setPosition({
      x: rect.left + rect.width / 2,
      y: rect.top - 8
    });
    setVisible(true);
  };

  const categoryColors = {
    technical: '#6366F1',    // Indigo for tech terms
    workplace: '#10B981',    // Emerald for workplace culture terms
    business: '#F59E0B',     // Amber for business/finance terms
    process: '#8B5CF6',      // Violet for process terms
  };

  return (
    <>
      <span
        ref={ref}
        className="jargon-term"
        style={{
          borderBottom: `1.5px dashed ${categoryColors[category]}`,
          cursor: 'help',
          color: 'inherit',
        }}
        onMouseEnter={handleInteraction}
        onMouseLeave={() => setVisible(false)}
        onTouchStart={handleInteraction}  // Mobile long-press handled separately
      >
        {children}
      </span>

      <AnimatePresence>
        {visible && (
          <motion.div
            className="jargon-popup"
            style={{
              position: 'fixed',
              left: position.x,
              top: position.y,
              transform: 'translate(-50%, -100%)',
              zIndex: 9999,
            }}
            initial={{ opacity: 0, y: 4, scale: 0.97 }}
            animate={{ opacity: 1, y: 0, scale: 1 }}
            exit={{ opacity: 0, y: 2, scale: 0.97 }}
            transition={{ duration: 0.12 }}
          >
            {/* Category pill */}
            <div className="jargon-category" style={{ color: categoryColors[category] }}>
              {category}
            </div>

            {/* Term */}
            <div className="jargon-term-header">{term}</div>

            {/* Definition */}
            <div className="jargon-definition">{definition}</div>

            {/* Example in context */}
            {example && (
              <div className="jargon-example">
                <span className="jargon-example-label">In context: </span>
                "{example}"
              </div>
            )}

            {/* "Heard it, got it" button — logs that this term was encountered */}
            <button
              className="jargon-ack-btn"
              onClick={() => {
                logTermEncountered(term);  // Used in the learning analytics
                setVisible(false);
              }}
            >
              Got it ✓
            </button>
          </motion.div>
        )}
      </AnimatePresence>
    </>
  );
}
```

**Step 4 — Dynamic Jargon Generation:**
When a character uses a term not in the library, make a background Claude API call:

```javascript
async function generateJargonEntry(term, context) {
  const response = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: "claude-sonnet-4-20250514",
      max_tokens: 200,
      messages: [{
        role: "user",
        content: `Generate a jargon tooltip entry for the term "${term}" as it appears
        in this workplace context: "${context}"
        
        Target audience: first-generation Indian graduate, no corporate experience.
        Tone: friendly, plain English, zero condescension.
        
        Return ONLY valid JSON:
        {
          "category": "technical|workplace|business|process",
          "definition": "one sentence, plain English, no jargon in the definition itself",
          "example": "a realistic usage sentence from an Indian workplace"
        }`
      }]
    })
  });
  const data = await response.json();
  return JSON.parse(data.content[0].text);
}
```

## MODE 2: THE WAR ROOM

### Concept

A real-time crisis unfolds on a single screen. The learner is not chatting — they are
**triaging**. Multiple information streams arrive simultaneously. A clock is ticking.
The learner must: assess information, make decisions, assign priorities, communicate
to different people in parallel, and prevent catastrophic escalation.

This is the closest simulation to how it actually feels when production breaks on a Friday.

### The Situation Board (Left Panel)

- 3–5 "incident cards" showing different simultaneous problems
- Each card has: title, severity bar (visual fill, color-coded), status label
  (Unknown / Investigating / Contained / Resolved), and owner (Unassigned or a name)
- The cards UPDATE as the simulation progresses based on learner's decisions
- If the learner ignores a card, its severity bar rises. If they address it, it drops.
- Cards glow red/amber depending on severity

### The Live Feeds (Center Panel)

4 columns of information arriving in real-time:

**Feed 1 — Slack Alerts (#alerts):**
Automated messages arriving from monitoring systems. Technical. Fast.
Example:
```
[09:47] ⚠ checkout-service: 500 error rate 12% (normal: 0.1%)
[09:49] ⚠ payment-processor: connection pool exhausted
[09:51] ✅ auth-service: responding normally
[09:53] ⚠ checkout-service: 500 error rate now 34% — escalating
```

**Feed 2 — Error Logs (simulated terminal output):**
Raw-looking log lines. Jargon tooltips active on every technical term.
Example:
```
ERROR [payment-service] ConnectionPoolTimeoutException:
  No connections available in pool (size: 10, in-use: 10)
  at PaymentProcessor.charge() line 247
  Attempted 3 retries, all failed
```

**Feed 3 — Messages (from humans):**
DMs from characters arriving unpredictably. Each needs a response.
Example:
```
Priya [PM]: Is checkout still broken? I have a client on the phone.
Client Vikram: Our campaign launched 10 min ago. 0 conversions. What's happening??
Kiran [Junior]: I think I know what the issue is?? should i try something??
Anjali [CTO]: Status update by 10am please. Short and clear.
```

**Feed 4 — Team Updates (internal broadcast):**
Other engineers posting updates. The learner must read these and adapt.
Example:
```
[Rajan, 09:52]: Looked at the logs. Connection pool issue in payment service.
               Config change deployed last night. Reverting now.
[Rajan, 09:55]: Revert deployed to staging. Testing.
[System, 09:58]: Rajan needs someone to verify the fix in staging — who can help?
```

### The Response Console (Right Panel)

5 action types, each opens a different interface:

**▸ Communicate:**
A small compose window. You choose: WHO (character selector), MEDIUM (DM / channel / email),
URGENCY (normal / urgent). You type 1–3 sentences. The JargonTooltip system is active.
The character responds in the Live Feed.

**▸ Escalate:**
A structured escalation form:
- Who are you escalating to? (dropdown: CTO / External vendor / On-call engineer)
- Current severity: (P0 / P1 / P2)
- Your read of the situation: (2-sentence text input)
- What you've already tried: (text input)
The system evaluates: is this the right person to escalate to? Is the timing correct?
Is the summary clear enough for a non-technical reader?

**▸ Assign:**
Drag-and-drop an incident card to a character's name. The character reacts based on:
their current load (shown as a small bar), their skill area, and whether the assignment
was communicated properly (if you assign without messaging, they don't know).

**▸ Delay / Hold:**
Explicitly decide that something is low enough priority to wait. This is a conscious
decision, timestamped. The diagnostic evaluates whether your delay decisions were correct.
(Choosing to delay a P0 = serious error. Choosing to delay a P2 while firefighting = smart.)

**▸ Decide:**
Some situations require a binary decision with significant consequences:
```
⚡ DECISION REQUIRED [09:56]
Rajan's revert is ready to deploy to prod.
Client campaign is live — every minute costs conversions.
BUT: the revert hasn't been tested for > 5 minutes yet.

[Deploy now — risk it]    [Wait for more testing]
[Ask Rajan to decide]     [Escalate to CTO]
```
The learner must decide. The decision has consequences. The system shows the consequence
after 30 seconds (whether production recovers, whether a new problem emerges).

### Clock and Pressure Mechanics

- A large countdown timer at the top. 15 minutes. Real-time.
- "Pressure indicators" that rise as time passes without action on critical items
- At 5 minutes remaining: a new problem appears (the "second wave")
- At 0 minutes: simulation ends. Unresolved incidents become diagnostic data.
- "Consequence moments" — at 3 key points, the simulation shows a 3-second consequence
  card: "Because you escalated to the CTO with a clear summary, she gave you authority
  to deploy the revert immediately." These feel like game feedback and are also diagnostic.

### War Room Diagnostic (unique dimensions)

Beyond the standard 8 dimensions, the War Room adds:
- **Crisis Composure:** Did you panic, freeze, or systematically work through it?
- **Triage Accuracy:** Did you correctly identify which fires to fight first?
- **Information Hygiene:** Did you read the feeds before acting, or act before reading?
- **Parallel Communication:** Could you maintain multiple conversations simultaneously?

## MODE 3: THE BRANCH ENGINE

### Concept

A cinematic, choose-your-own-path narrative. The learner doesn't type — they **choose**.
But unlike a quiz, the choices are nuanced. There is rarely an "obviously right" answer.
The consequences unfold over 3–5 turns. After completing the branch, the learner sees
a **consequence map** — a visual tree showing exactly how their choices cascaded,
what each fork would have led to, and what others chose.

This mode teaches decision-making under ambiguity without the pressure of real-time.
It is the most "reflective" mode. Good for learners who need to internalize WHY before HOW.

### The Screen

### Choice Design Principles

Each scenario has 4 choices. The choices are designed as follows:
- **A:** The "nice person" choice. Popular. Usually has hidden costs.
- **B:** The "direct" choice. Sometimes right, sometimes too blunt.
- **C:** The "professional" choice. Correct in most cases. Not obvious.
- **D:** The "avoidance" choice. Popular with anxious people. Usually has consequences.

Hovering over a choice shows a tooltip: **"This tests: Constructive Pushback"**
This removes the feeling of being "tricked" while still creating genuine decision-making.

### Consequence Display

After a choice, a 3-second consequence card plays before the next scene:

```

  your Friday deadline on KT-118. Priya was disappointed.
  The client asked where the bug fix was. You had to
  explain a missed commitment in your first week.

  💡 What the best performers did:
  Most chose C — committing while setting expectations.
  "I can add it, but I need to shift my KT-118 ETA to Monday.
  Is that okay with the client?" is one sentence that prevents
  three days of awkwardness.
  ──────────────────────────────────────────────────────
```

### The Consequence Map (Post-Scenario)

After all 5 scenes, the learner sees a visual tree:

```

  │   │   │   └── OUTCOME: Missed deadline. Damaged trust. Low escalation score.
  │   │   │

  │   │       └── OUTCOME: Trust partially preserved. Late, but professional.

          │

```

The map is built dynamically by Claude based on the learner's actual path + all
the alternative paths generated by the scenario. Interactive — the learner can
click "What if I had chosen B?" and see the alternative branch animate in.

## MODE 4: INBOX ZERO

### Concept

The most realistic, least glamorous, most universally useful simulation.

You arrive at your desk on Monday morning. Your inbox has 23 emails, 6 Slack messages,
3 calendar invites for today, and 1 missed call. Your first meeting is in 30 minutes.
You have a deadline at 2pm.

There is no right answer to "what do you do first." There is a set of principles that
experienced professionals use, and a set of traps that first-time workers fall into.
This mode observes which ones you use.

### The Screen

```

│                              │  ACTIONS on selected email:             │

│  Fri 6:23 PM                 │                                         │
│  [... 18 more]               │                                         │
└──────────────────────────────┴─────────────────────────────────────────┘
```

### The Email Types (designed for maximum diagnostic value)

The 23 emails are carefully designed across these categories. Each tests something specific:

**Urgent + Important (must handle now):**
- Priya's Saturday night "did you check prod?" — tests: does learner flag this immediately
  or treat it like a normal email?
- Client Vikram's follow-up on a Friday promise — tests: does learner understand that
  client follow-ups on open commitments are P1?

**Important + Not Urgent (must schedule):**
- Rajan's PR review request — tests: does learner acknowledge receipt even if they
  can't review right now?
- A meeting invite for Wednesday with pre-read attached — tests: does learner RSVP
  and actually open the pre-read, or ignore until Tuesday night?

**Urgent + Not Important to You (must delegate or decline):**
- A request from another team to help debug something that's not your stack
  Tests: can the learner say "this isn't mine, but here's who can help"?
- HR compliance training reminder (deadline: end of this week)
  Tests: is the learner overloaded enough to defer a 30-min task that has a deadline?

**Not Urgent + Not Important (should archive/ignore):**
- Newsletter
- A "great meeting!" reply-all from a meeting you weren't in
- A vendor pitch disguised as an opportunity
  Tests: can the learner identify and triage low-value emails quickly?

**Emotionally Charged (requires careful handling):**
- A passive-aggressive email from a colleague about a missed deadline from last week
  Tests: does the learner reply defensively? Avoid it? Handle it professionally?
- A compliment email from a senior leader
  Tests: does the learner know how to acknowledge it gracefully? (Many Indian graduates
  are so unused to receiving praise they don't know how to respond.)

**Traps:**
- An email that looks urgent but is actually a forwarded thread you were CC'd on
  Tests: does the learner read the whole thread before replying, or reply to the
  top email without understanding context?
- An email asking you to make a decision that isn't yours to make
  Tests: does the learner recognize they don't have authority here and route accordingly?

### Inbox Zero Diagnostic (unique)

- **Triage Accuracy Score:** Did you correctly sort emails into urgent/important/neither?
- **Response Quality:** Were your replies clear, appropriate in length, and professional?
- **Delegation IQ:** Did you correctly identify emails that weren't yours to handle?
- **Time Management Signal:** Did you spend too long on low-priority emails?
- **Emotional Regulation:** How did you handle the passive-aggressive email?

### 30-Minute Timer Mechanic

The timer creates the same pressure as a real Monday morning. At 9:30 AM (T+27 min):
a notification: "🗓 Your standup starts in 3 minutes." The learner must stop and
transition. Any unprocessed emails remain — the diagnostic shows what they left.

At the end, the diagnostic shows:
- A visual breakdown of how the learner spent their 27 minutes
- A "what experienced professionals do" comparison (benchmark behavior)
- The 3 most important emails that weren't handled optimally
- A recommendation: "Next time, start with any email that is urgent + has a human
  waiting on your response. Process those first, then batch everything else."

## MODE 5: THE ARENA (NEGOTIATION)

### Concept

One table. One counterpart. One deal. A structured negotiation simulation where
the learner can SEE their relationship health and deal value changing in real time
as they make moves. This is not just chatting — it has game mechanics visible to the learner.

### The Screen

```
┌─────────────────────────────────────────────────────────────────────────┐
│  🤝  NEGOTIATION ARENA  ·  Scene: Rate Discussion with Nandini (Client) │
├───────────────────────┬────────────────────────────┬────────────────────┤
│  YOUR POSITION        │  THE NEGOTIATION            │  HER POSITION      │
│                       │                             │                    │
│  Your rate: ₹40,000   │  [Conversation thread       │  Budget: ₹25,000   │
│  Your floor: ₹30,000  │   between learner and       │  (YOU DON'T KNOW   │
│  (hidden from her)    │   Nandini, Slack style]     │  her floor yet)    │
│                       │                             │                    │
│  BATNA: You have 2    │  Nandini: "We love your     │  BATNA: She has    │
│  other clients.       │  work. Our budget is        │  one other vendor  │
│  You can walk away.   │  tight — ₹25k max?"         │  at ₹28,000 but   │
│                       │                             │  lower quality.    │
│                       │  [Your response here]       │  (YOU DON'T KNOW   │
│                       │                             │  this yet)         │
├───────────────────────┴────────────────────────────┴────────────────────┤
│  LIVE METRICS                                                            │
│                                                                          │
│  Relationship Health:  ████████░░  78%   (changes with your tone)       │
│  Deal Probability:     ██████░░░░  60%   (changes with your moves)       │
│  Your Leverage:        █████░░░░░  50%   (revealed by your BATNA usage)  │
│  Information Asymmetry: ████░░░░░  45%   (decreases as you share/reveal) │
│                                                                          │
│  AVAILABLE MOVES:                                                        │
│  [Anchor high]  [Make concession]  [Ask a question]  [Reveal BATNA]     │
│  [Reframe value]  [Propose package]  [Request time]  [Walk away]        │
└──────────────────────────────────────────────────────────────────────────┘
```

### The Negotiation Mechanics

**Available Moves (each is a button + compose window):**

- **Anchor High:** Start with your number, confidently. Your compose window opens.
  The system shows a "how to anchor" tooltip: "Name a number with reasoning, not apology.
  'My rate for this scope is ₹40,000. That includes X, Y, Z.'"

- **Make Concession:** Reduce your ask, but extract something in return.
  The window shows: "Never give without getting. 'I can move to ₹35,000 if we
  can agree on monthly payment instead of milestone-based.'"

- **Ask a Question:** Information is power. "What does your timeline look like?
  What other options are you evaluating?" The AI character answers — sometimes
  revealing information you can use.

- **Reveal BATNA:** "I have two other projects I'm considering — this helps me
  understand how to prioritize." This increases your leverage (visible in the metric)
  but risks damaging the relationship if done clumsily.

- **Reframe Value:** Shift the conversation from price to value.
  "Let's talk about what this deliverable is worth to your launch. If it drives
  ₹5 lakh in revenue, ₹40,000 is 0.8% of that."

- **Propose Package:** Bundle different elements to create new variables.
  "What if I do ₹30,000 for this project, and you give me a referral to two
  other teams in your company?"

- **Request Time:** "Can I get back to you by Thursday? I want to review
  the full scope before confirming." Tests: can the learner use silence strategically?

- **Walk Away:** "I think we have different budgets in mind. I'd be happy to
  revisit if your situation changes." Tests: can the learner walk away?
  (Most Indian graduates can't. They've been taught to accommodate.)

### The Live Metrics

The four bars update visibly after every move. The learner can see:
- Relationship Health dropping when they're too aggressive
- Deal Probability rising when they reframe value effectively
- Their Leverage increasing when they mention their BATNA
- Information Asymmetry decreasing as the negotiation progresses

This visible feedback is the key innovation — negotiation is usually opaque.
KarmaLoop makes the game mechanics visible so learners can connect their moves
to their effects in real time.

### The Hidden Information System

At the start, the learner knows their own position. They don't know:
- The client's true budget floor
- Whether the client has alternatives
- What the client values most (price? quality? speed? exclusivity?)

Through **asking questions** (the "Ask a Question" move), they can surface hints.
The AI character reveals information realistically — a little at a time, sometimes
evasively, sometimes more than intended.

At the end, the post-negotiation debrief reveals the client's full position,
so the learner can see how much information was available and how much they left
on the table by not asking.

### Audience Body Language System

Each audience member has a visible emotional state indicator that changes in real time
based on what the learner says and how they say it:

**States:** 😊 Engaged → 😐 Neutral → 🤔 Skeptical → 😑 Disengaged → 🙋 Has a question

**What changes states:**
- Too much jargon with the CEO → CEO becomes Disengaged
- Strong data cited → Divya becomes More Engaged
- "I'm not sure" without context → Aakash becomes Skeptical
- Confident, specific answer to a hard question → all states improve
- Defensiveness when challenged → Divya becomes Disengaged
- Conceding a valid point gracefully → all states improve slightly

The engagement meter is the aggregate. If it drops below 40%, the CEO looks at her phone.
If it drops below 20%, Aakash says "maybe we should come back to this."

### Interruption System

At 3 points during the presentation, a character interrupts:

**Early interruption (Aakash, 2 minutes in):**
"Wait — can you explain why you made this navigation choice? I would have done it
completely differently."

**Mid-presentation (CEO, midpoint):**
"This feels cold. Our users are patients — they're anxious. Doesn't this design
make that worse?"

**End interruption (Divya, just before wrap-up):**
"Before you close — the research finding on screen 2. What was your sample size?
I want to know if we can draw conclusions from this."

The learner must handle each interruption. Their response options:
- **Address directly:** Type a response. The character reacts.
- **Acknowledge and park:** "Great point — can I finish this slide and come back to it?"
- **Ask a clarifying question:** "What specifically about the navigation concerns you?"
- **Show supporting data:** Opens the research document to find the relevant finding.

### Learner Input Methods

**Typing their spoken words:** The primary input. Learner types what they would say.
The system evaluates: length (too long = losing the room), clarity, jargon level
(too technical for CEO = engagement drops), confidence markers ("I think" vs. "The data shows").

**Navigation controls:** Next/Back slide, Pause for questions.

**Action buttons:**
- "Show data" — opens a data panel with a chart or research finding the learner can
  reference. Using data at the right moment boosts engagement.
- "Reference user research" — quotes a specific research finding. Powerful for designers.
- "Address objection" — opens a structured response template: "I hear your concern about X.
  Here's why we made this decision: [reason + data]. What would change your mind?"
- "Acknowledge and move on" — gracefully concedes a point without losing momentum.

### Stage Diagnostic (unique dimensions)

- **Narrative Clarity:** Did your presentation tell a story, or list facts?
- **Audience Awareness:** Did you adjust your language per person? (CEO ≠ Aakash)
- **Defense Quality:** When challenged, did you use data, intuition, or defensiveness?
- **Pacing:** Did you rush, drag, or maintain appropriate tempo?
- **Recovery:** When the CEO said "this feels cold" — how did you recover?

### The "Weak Dimension Drill" (New in V3)

When a learner's diagnostic shows a persistent gap (score ≤ 2 across 2+ sessions),
KarmaLoop offers a **focused drill** — a 5-minute micro-simulation targeting exactly
that dimension.

Example: Learner consistently scores 1–2 on "Honest Estimation."

```
  FOCUSED DRILL: Honest Estimation

  You've scored "Developing" on this dimension in 2 sessions.
  Let's drill it specifically.

  The drill: 8 short scenarios, each with a variation of the same situation.
  Someone asks you for an ETA you don't have yet. You respond.
  Claude evaluates: did you give a specific, bounded, honest answer?

  [ Start 5-minute drill ]
```

Each scenario is 30 seconds. The learner types their response. Claude evaluates
immediately with a green check or red cross and the specific phrasing they should
have used. Rapid, repetitive, targeted — like a pianist practicing scales,
not a concert.

### Cross-Mode Synthesis Diagnostic

After completing 3+ modes, KarmaLoop generates a **Cross-Mode Synthesis Report**.
This is the most sophisticated diagnostic in the system.

```
SYNTHESIS PROMPT (Claude):

You have observed this learner across {{N}} simulation modes:
Mode 1 diagnostic: {{DIAGNOSTIC_1}}
Mode 2 diagnostic: {{DIAGNOSTIC_2}}
Mode {{N}} diagnostic: {{DIAGNOSTIC_N}}

Generate a cross-mode synthesis that identifies:

1. PERSISTENT PATTERNS: Behaviors that appear consistently across modes,
   regardless of context. These are deeply ingrained habits.
   Example: "In 3 of 4 modes, you avoided proactive communication until
   you were directly asked for a status update."

2. CONTEXT-DEPENDENT STRENGTHS: Things you do well only in certain contexts.
   Example: "Your constructive pushback is strong in text-based communication
   (Modes 1, 4) but drops significantly under time pressure (Mode 2)."

3. GROWTH TRAJECTORY: Are you improving, plateauing, or regressing on each dimension?
   Based on session timestamps and scores.

4. THE ONE THING: If this learner could only work on one dimension before their
   next job interview or first week at work, what should it be? And exactly how?

5. READINESS SIGNAL: On a scale of 1–10, how job-ready does this learner appear
   for their target career track? What specific milestone would move the needle most?

Output: structured JSON with these 5 sections. Be specific. Be honest.
Reference exact moments from exact modes. This is mentorship at scale.
```

### Mode Selector Screen

- Dark background, 6 cards in 2×3 grid (or 3×2 landscape mobile)
- Card dimensions: 200×180px desktop
- Each card has: large icon (48px), mode name (20px Inter 600), description (13px secondary),
  duration tag (small amber pill), tone tag (small violet pill)
- Completed modes: emerald top border, checkmark overlay, score shown
- Active/recommended: subtle Indigo pulsing border
- "Recommended next" badge: floating above the recommended card
- Transition: clicking a card triggers a cinematic slide transition to that mode

### War Room Design

- 3-column layout, dark red accent for critical severity (#1A0505 background tint)
- Situation Board cards: each has a glowing severity border (red = critical)
  The glow intensity matches severity percentage
- Live feeds: 4-column scrolling feed. New items slide in from bottom of each column.
  Each feed has a small header (⚠ Alerts / 📟 Logs / 💬 Messages / 👥 Team)
- Decision cards: full-width, amber background (#1A1400), large text
  The 4-choice buttons are large (full-width), visually weighted
- Consequence reveal: 3-second fullscreen overlay, dark with colored result text
- Clock: large, top-center, red when < 5 minutes remaining. Pulses at < 2 minutes.

### Branch Engine Design

- Single-panel focus. No distractions.
- Narrative card: large, centered, #141414, soft inner shadow, 20px body text
  Character message inside the card: Indigo left border, italic
- 4 choice buttons: 2×2 grid, equal sizing, #1E1E1E bg, hover = Indigo border
  Each button has: letter label (A/B/C/D in circle), choice text, hover tooltip
- Consequence overlay: slides up from bottom, takes 40% of screen height
  Color-coded: red = negative consequence, amber = mixed, green = positive
- Consequence map: full-screen after all scenes. Rendered as a visual tree.
  Visited path in Indigo. Unvisited alternatives in gray (explorable on click).
  Animated: paths draw themselves from start to end node.

### Inbox Zero Design

- Two-panel layout: email list (left, 35%) + reading pane (right, 65%)
- Email list items: sender name (bold), subject preview (secondary), time tag
  Priority indicators: red dot (urgent), no dot (normal), gray (likely unimportant)
  Hover: full-width Indigo left border
  Selected: #1A1A2E background
- Reading pane: email content with JargonTooltip on all technical/corporate terms
  Action buttons: horizontal bar below email content, icon + label buttons
- Timer: top right, large, turns amber at 10 minutes, red at 5 minutes
- Progress bar: "Emails processed: 3 / 23" with a fill animation
- Draft window: slides up from bottom-right when "Reply" is selected
  Takes 45% of reading pane height. Auto-focuses cursor.

### Negotiation Arena Design

- 3-column: Your Position (left) / Conversation (center) / Her Position (right)
  Your position info: always visible. Her position: partially hidden/revealed over time.
- Live metrics: 4 progress bars below the conversation, labeled, color-coded
  Bars animate smoothly after every exchange (0.4s ease transition)
- Move buttons: grid of 8 at the bottom, icon + label, #1E1E1E background
  Active/clicked: Indigo background, white text
  "Walk Away" button: red tint, separated slightly from others
- Compose window: opens on top of conversation when a move is selected
  Shows the move name as a label, a "how to do this" tooltip, and a text area
- Post-negotiation reveal: full-screen split showing both parties' actual positions
  Animated: her hidden info slides in from right

### The Stage Design

- Presentation area: top 45% of screen, dark with slide/prototype content inside
- Audience row: 3 character cards below the presentation area
  Each card: avatar (48px), name, role, and a live emotional state badge
  Badge uses emoji + text: "😐 Listening" / "🤔 Skeptical" / "🙋 Has a question"
  Badge changes animate with a 300ms fade transition
- Engagement meter: single bar, full-width, color shifts (green > 60%, amber 40-60%, red < 40%)
- Interruption event: character card zooms slightly (scale 1.1), their badge becomes
  🙋 Has a question, then their speech appears in a chat bubble above their card
- Learner console: bottom 20% of screen, fixed, contains: text input, navigation controls,
  action buttons in a row. Always accessible.
