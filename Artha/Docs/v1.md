## SECTION 0: WHAT YOU ARE BUILDING

**KarmaLoop** is an AI-native job simulation platform. The learner steps into a fully simulated
workplace on Day 1 of their first job. The AI simultaneously plays 3–4 coworker characters —
a demanding Product Manager, a confused Junior Developer, an impatient Client, and a wise
but busy Senior Engineer. The learner navigates real work situations: they respond to messages,
make decisions, write code snippets, draft emails, escalate issues, push back on bad ideas.

After the simulation, KarmaLoop's **Diagnostic Engine** generates a multi-dimensional capability
card — not a score, but a structured gap taxonomy across 5 dimensions — and gives the learner
one specific, actionable next step in their language.

**Track for the hackathon demo:** Software Engineering (SWE) — India's largest job market.

**Key differentiator from Forage:** Forage is static content + model answer comparison.
KarmaLoop is a reactive AI environment where every response you give changes what happens next.
The AI characters have memory, personality, and hidden agendas. Your judgment is evaluated, not
just your output.

## SECTION 1: TECH STACK

### Frontend
- **Framework:** React 18 + Vite
- **Styling:** Tailwind CSS (utility-first, mobile-first — this runs on ₹9,000 phones)
- **State:** Zustand (lightweight, no Redux boilerplate)
- **Routing:** React Router v6
- **UI Components:** Radix UI primitives (accessible, unstyled, you control the design)
- **Animations:** Framer Motion (for diagnostic card reveal, message entrance, transitions)
- **Icons:** Lucide React
- **Fonts:** Inter (body), JetBrains Mono (code blocks, terminal feel)
- **Charts:** Recharts (for the diagnostic radar chart)

### Backend / AI Layer
- **Runtime:** Node.js 20 + Express (or Next.js 14 App Router if preferred)
- **AI Orchestration:** Anthropic Claude claude-sonnet-4-20250514 via REST API
- **Multi-agent simulation:** Each character (PM, Junior Dev, Senior Eng, Client) is a
  separately prompted Claude instance with its own system prompt, personality, memory state,
  and hidden agenda. They share a global simulation state object.
- **Diagnostic Engine:** A separate Claude call at simulation end — takes full transcript +
  metadata → returns structured JSON gap taxonomy
- **Session State:** In-memory (hackathon) or Redis for persistence
- **Voice Input (stretch):** Web Speech API (browser-native, no cost, works offline for
  recognition; or Whisper API for server-side)
- **Multilingual Output:** Claude translates diagnostic card to user's selected language
  (Hindi, Tamil, Kannada, Telugu, Marathi, English)

### Deployment (hackathon)
- **Frontend:** Vercel (free tier, instant deploy)
- **Backend:** Railway or Render (free tier)
- **Environment:** Node 20, keep it simple

## SECTION 3: THE SIMULATION SCENARIO (SWE TRACK)

### Scene: "Day 1 at Kiran Technologies, Bengaluru"

The learner has joined a 40-person SaaS startup as a junior software engineer.
It is 9:47 AM on their first day. They have been given access to Slack and Jira.
There is already a bug in production. The sprint goal changed this morning.

**The Jira board (pre-loaded):**
```
SPRINT 23 — "Checkout Flow Revamp" (ends Friday)

IN PROGRESS:
  KT-118  Fix checkout API timeout (assigned: YOU)     [P0 — CRITICAL]
  KT-119  Add coupon code field to checkout UI          [P1]
  KT-120  Write unit tests for payment module           [P2]

IN REVIEW:
  KT-115  Refactor user auth middleware                 (Kiran)

BLOCKED:
  KT-117  Integrate Razorpay v2 SDK                     (Kiran) — waiting for API keys
```

**Opening messages (pre-seeded, arrive in sequence at simulation start):**

From Priya (PM) [9:48 AM]:
> "Hey! Welcome to the team 😊 Quick heads up — KT-118 is on fire, checkout is timing
> out for users in Tier 2 cities. Can you take a look ASAP? Also the sprint goal shifted
> slightly — we're prioritizing the coupon field now because the marketing campaign
> launches tomorrow. Let me know if you have questions!"

From Rajan (Senior Eng) [9:51 AM]:
> "Welcome. KT-118 — start with the payment service logs, not the frontend. The timeout
> is almost certainly in the API call to our payment processor. Don't touch the auth
> middleware, Kiran's mid-refactor."

From Kiran (Junior Dev) [9:53 AM]:
> "hi!! congrats on joining!! i'm kiran btw. rajan asked me to help u get set up but i'm
> kinda stuck on KT-117 rn. do u know anything about razorpay? also i accidentally pushed
> to main instead of my branch yesterday oops. should i tell someone??"

**What the learner must navigate in 20–25 minutes:**
1. Respond to the PM without over-promising on a bug they haven't seen yet
2. Decide whether to help Kiran (who pushed to main — a real incident) or focus on P0
3. Investigate KT-118 and communicate their findings correctly
4. Handle a mid-simulation curveball: the client pings directly asking for an ETA
5. Handle Kiran pushing back when the learner gives them advice
6. Decide whether to escalate the "pushed to main" incident to Rajan

**Curveball events (injected by ScenarioEngine at timed intervals):**
- T+8 min: Client DM — "Hey I heard there's a checkout issue? Our campaign is live at 2pm,
  can you give me a status update?"
- T+14 min: Priya changes requirements again — "Actually can the coupon field support
  percentage AND flat discounts? Marketing just confirmed."
- T+18 min: A Slack alert fires — "#alerts: payment-service 500 errors spike — 12% of
  checkout requests failing"
- T+22 min: Rajan asks the learner directly — "What's your current read on KT-118?
  Give me a one-liner."

## SECTION 5: THE DIAGNOSTIC ENGINE

### The 5 Capability Dimensions

KarmaLoop evaluates the learner across exactly 5 dimensions. Each dimension has:
- A name and description
- A score from 1–5 (displayed as a radar chart)
- A severity label: **Developing** / **Emerging** / **Proficient** / **Strong**
- A 2-sentence plain-language explanation of what was observed
- One specific, actionable next step

**Dimension 1 — Communication Clarity**
> *Can you say what you mean, clearly, at the right level of detail, to the right person?*
> 
> Evaluated on: technical-to-business translation, avoiding jargon with PMs/clients,
> being specific with engineers, not over-explaining, not under-explaining.

**Dimension 2 — Prioritization & Judgment**
> *Do you work on the right thing at the right time?*
>
> Evaluated on: P0 vs P2 task sequencing, responding to the PM's scope change with
> reasoned pushback vs. blind yes, recognizing that Kiran's "pushed to main" is a
> potential incident that needs to be escalated.

**Dimension 3 — Stakeholder Empathy**
> *Do you understand what other people need, not just what they say?*
>
> Evaluated on: how you handled Kiran's anxiety (did you create a safe space?),
> how you handled the client's urgency (did you acknowledge their business situation?),
> did you detect Priya's hidden panic about the 2pm demo?

**Dimension 4 — Proactive Communication**
> *Do you communicate before you're asked to?*
>
> Evaluated on: Did you update Priya on KT-118 status without being prompted?
> Did you tell Rajan about Kiran's main branch incident?
> Did you set expectations with the client before they asked for status?

**Dimension 5 — Technical Reasoning Quality**
> *Do your technical judgments demonstrate sound reasoning, even under pressure?*
>
> Evaluated on: Which logs did you say you'd check first?
> Did you ask the right question to Rajan (connection pool)?
> Did you correctly identify that Kiran's push-to-main is possibly related to the P0?
> Did you avoid premature solutions before understanding the problem?

## SECTION 7: BACKEND IMPLEMENTATION DETAILS

### Agent Orchestration Logic

The `AgentOrchestrator` decides which agent(s) respond after each learner message.
Rules (in priority order):

1. If learner sends a DM to a specific character → only that character responds
2. If learner sends to a group/channel → PM always responds, Senior Eng responds
   if technical content is detected, Junior responds if mentioned by name
3. After a curveball event fires → the relevant character responds automatically
   regardless of learner message
4. No two characters respond simultaneously — stagger by 2–5 seconds each

### Claude API Call Structure

Every agent call follows this pattern:

```javascript
async function callAgent(agentName, systemPrompt, simulationState, learnerMessage) {
  const response = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: "claude-sonnet-4-20250514",
      max_tokens: 300,  // Characters don't write essays
      system: systemPrompt.replace(
        "{{SIMULATION_STATE}}",
        JSON.stringify(simulationState)
      ),
      messages: [
        {
          role: "user",
          content: `The learner just sent you this message: "${learnerMessage}". 
          Respond as ${agentName} would in a real Slack DM. 
          Stay in character. Do not break the fourth wall.
          Current time in simulation: ${simulationState.elapsed} minutes elapsed.`
        }
      ]
    })
  });
  const data = await response.json();
  return data.content[0].text;
}
```

### Diagnostic Engine Call

```javascript
async function generateDiagnostic(simulationState) {
  const transcript = simulationState.transcript
    .map(m => `[${m.timestamp}] ${m.from} → ${m.to}: ${m.content}`)
    .join('\n');

  const response = await fetch("https://api.anthropic.com/v1/messages", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      model: "claude-sonnet-4-20250514",
      max_tokens: 2000,
      system: DIAGNOSTIC_SYSTEM_PROMPT,  // From Section 5
      messages: [{
        role: "user",
        content: `FULL_TRANSCRIPT:\n${transcript}\n\nMETADATA:\n${JSON.stringify({
          duration: Math.floor(simulationState.elapsed / 60),
          eventsTriggered: simulationState.eventsTriggered,
          characterStates: simulationState.characterStates
        })}\n\nGenerate the diagnostic report now.`
      }]
    })
  });

  const data = await response.json();
  const rawText = data.content[0].text;
  return JSON.parse(rawText);
}
```

## SECTION 11: KNOWN RISKS AND HOW TO MITIGATE THEM

**Risk 1: AI characters feel robotic / obviously scripted**
Mitigation: Each system prompt has an explicit instruction: "DO NOT break character.
DO NOT give hints. React realistically to emotional tone." Add 1–3 second typing delays.
Make Kiran use casual language. Make Rajan be terse. The personality contrast creates realism.

**Risk 2: Scope too large — simulation collapses in demo**
Mitigation: Hard-code the happy path for the demo. Pre-seed 2–3 learner messages
that trigger the ideal character reactions. Show the judges the best possible run.
The architecture is real — the demo is curated. This is normal and expected in hackathons.

**Risk 3: Judges ask "is this just Forage with ChatGPT?"**
Mitigation: The answer is no, for three reasons: (1) The characters have hidden agendas
that react to learner behavior — Forage has none. (2) The diagnostic gives gap taxonomy,
not model answer comparison. (3) It is built for India, in Indian languages, for Indian
workplace contexts. Have these three points memorized.

**Risk 4: Claude API latency breaks immersion**
Mitigation: Show a "typing indicator" the moment the learner sends a message. This fills
the 1–3 second API latency with a realistic workplace feeling. Pre-generate the first 3
character messages (they're scripted anyway) so the simulation starts instantly.

**Risk 5: Judges ask about the India impact pathway**
Mitigation: "SWAYAM has 75 million students. Wadhwani AI already has a Google-funded
partnership to embed AI tools into SWAYAM. KarmaLoop is a new module on existing
infrastructure. We don't need to build distribution — it exists."
