# ARTHA V2 — THE COMPLETE BUILD PROMPT
## Wadhwani AI Hackathon · Problem 3.2 · Job Readiness & Entrepreneurial Skills
### Expanded: Multi-Track, ML-Personalized, Full Workplace Reality Simulation
### For: Claude Code (full-stack) + Claude Design (UI/UX)

---

> **THE NORTH STAR — READ BEFORE ANYTHING ELSE**
>
> India has an epidemic. Not of unskilled people. Of **invisible skill blindness.**
> 42.6% of Indian graduates are unemployable — not because they can't code, design, or
> analyse data, but because nobody ever showed them what the job *actually* looks like
> from the inside. What does it feel like when your manager gives you a vague brief?
> When your client goes silent for 3 days? When a senior tells you your work is wrong
> in front of the whole team? When you disagree with a decision that's already been made?
>
> Every Indian engineering and commerce college teaches the *content* of a job.
> Nobody teaches the *texture* of a job. The awkward standup where you have to
> admit you're blocked. The email where you have to ask for a deadline extension
> without sounding incompetent. The moment where you push back on a bad brief
> because you know it'll fail — and then navigate the political fallout.
>
> **KarmaLoop puts you inside that texture. Not as a lesson. As an experience.**
>
> Wadhwani AI's Alpan Raval: *"Impact-first, not research-first. We work backward
> from the potential impact."* Every feature in this product must pass one test:
> does it make a first-generation graduate from Lucknow more employable tomorrow?
> If not, cut it.

---

## WHAT IS NEW IN V2 (vs V1)

V1 was a strong SWE simulation. V2 is a **multi-career platform** with:

1. **Career selection onboarding** — 8 career tracks, the learner picks their aspiration
2. **ML-powered scenario personalization** — Claude adapts scenario difficulty,
   character intensity, and curveball timing based on the learner's self-assessed profile
3. **Massively expanded simulation elements** — not just tickets and chat, but:
   emails, standups, design reviews, incident calls, client negotiations, code reviews,
   data dashboards, document editors, and a verbal briefing interface
4. **The Invisible Skills Layer** — 12 micro-competencies that no college in India teaches,
   deliberately engineered into every simulation
5. **Entrepreneurial spirit track** — for those who want to start something
6. **Longitudinal learning arc** — 3 difficulty levels per track, diagnostic tracks
   improvement across sessions

**The gap:** Everyone knows DSA. Nobody teaches how to admit you're blocked in a standup.

**Scenario:** Day 3 at Kiran Technologies, Bengaluru. P0 bug. PM who just changed
the sprint goal. Senior in back-to-back meetings. Junior who pushed to main.

**Characters:** Priya (PM, impatient, friendly, has a hidden 2pm deadline),
Rajan (Senior Eng, terse, time-poor, respects initiative),
Kiran (Junior Dev, anxious, made a mistake, needs psychological safety to admit it),
Vikram (Client, anxious, doesn't understand tech, understands deadlines)

**Simulation Elements:** Jira board, Slack channels, GitHub PR review,

- Kiran pushed to main (escalation vs. psychological safety)

## SECTION 1: THE INVISIBLE SKILLS LAYER (12 MICRO-COMPETENCIES)

> These are the things no college in India teaches.
> Every simulation must hit at least 4 of these. Characters are designed to create
> these moments naturally — learners don't know they're being tested on them.

1. **Honest Estimation** — "I need 20 minutes to assess this. I'll update you by 10:15."
   Not "sure, no problem" and not "I have no idea."

2. **Asking for Help Without Seeming Incompetent** — "I've tried X, Y, Z and I'm stuck.
   Do you have 10 minutes?" — not asking before trying, not never asking.

3. **Handling Ambiguous Briefs** — Asking clarifying questions before starting.
   Not guessing. Not paralysis.

4. **Escalation Judgment** — Knowing when to solve it yourself, when to loop in
   your manager, when to alert the whole team. This is the highest-leverage
   invisible skill and almost universally untrained.

5. **The Pushback Art** — Disagreeing with a decision that's already been made.
   Respectfully. With evidence. Without burning the relationship.

6. **Proactive Status Communication** — Telling stakeholders where things stand
   before they ask. "Update: found the issue, working on fix, ETA 2 hours."

7. **Receiving Critical Feedback Without Ego** — Processing harsh code review comments,
   bad design feedback, wrong analysis — without defensiveness or collapse.

8. **Cross-Functional Translation** — Same technical issue, completely different
   framing for a PM vs. a developer vs. a CEO. Each audience needs different language.

9. **The No** — Saying no to scope creep without losing the relationship.
   Saying no to a senior without causing offense. Saying no when you're at capacity.

10. **Documenting While Doing** — Investigation logs. Post-mortems. Decision records.
    This is the invisible work that separates trusted professionals from those who are
    "hard to manage." Most graduates solve the problem and don't document.

11. **Managing Up** — Giving your manager enough information that they can protect
    and support you. Updating before they ask. Being predictable.

12. **Self-Advocacy** — Asking for a deadline extension before you miss it.
    Taking credit for work that went well. Flagging when you're overwhelmed before
    you collapse. Indian graduates are trained not to seem demanding. This costs them enormously.

## SECTION 3: SIMULATION ELEMENT LIBRARY

Each track activates a subset of these elements. All share the same base chat interface.

**A. Messaging (All tracks)**
WhatsApp/Slack-style threaded DMs per character + group channels.
Typing indicator (animated dots, 2–5 seconds before every reply).
Message timestamps advance in real time. Unread badges create urgency.

**B. Email Inbox (SWE, Cybersecurity, Marketing, Freelancer)**
Pre-loaded inbox with 1–2 emails that arrived "before" the sim started.
Learner can compose emails — diagnostic evaluates subject line, appropriate
formality, correct recipients, clarity. An email from the client at 7 AM
creates a "I should have responded earlier" moment.

**C. Ticket / Task Board (SWE, Cybersecurity, Data Science)**
Kanban: Backlog / In Progress / In Review / Done.
P0/P1/P2 badges. P0 ticket pulses in red. Learner moves tickets by describing actions.
Board updates dynamically as sim progresses. One assigned ticket starts OVERDUE.

**D. Document / Report Interface (Cybersecurity, Data Science, UX, Pentesting)**
Split panel: left = document draft with structured fields, right = chat.
Characters can comment on specific sections. Learner fills in real time.
Formats: Incident Report / Slide Outline / Pentest Finding / Design Review Notes.

**E. Data Dashboard (Data Science, Marketing, Cybersecurity)**
Simulated analytics panel with charts. Deliberate ambiguity (missing metric,
outlier data). Learner interacts by describing observations. "Run query" input.
Dashboard updates mid-simulation with new data that changes the picture.

**F. Design Review Interface (UX Track)**
4 static prototype screens in a viewer frame. Live comment thread where characters
add feedback in real time. "Research callout" button: learner quotes user research
findings to defend decisions. Timeline widget for trade-off decisions.

**G. Verbal Briefing Call Interface (Cybersecurity, Pentesting, Founder)**
"Call" button simulates a 3-minute verbal update to CISO / investor.
Learner types their spoken words as dialogue. Character responds to content AND tone.
Time limit countdown visible. Tests concision, confidence, business-language translation.

**H. Standup Bot (SWE)**
At T+10 min, bot posts: "Standup! Yesterday / Today / Blockers"
Diagnostic evaluates: specificity, honesty about blockers, appropriate length.
If skipped, PM notices and asks why.

**I. Code Review Interface (SWE)**
Diff-style view. Senior Rajan posts 3–4 inline comments — one constructive,
one dismissive, one factually wrong. Learner responds to each.
Diagnostic evaluates: did they push back on the wrong comment with evidence?

**J. Negotiation Interface (Freelancer)**
Structured back-and-forth on rate, scope, timeline. 3 rounds. Each character
has a BATNA hidden in their system prompt. Walking away can be the right answer.

**K. Cold Email / Pitch Composer (Freelancer, Founder)**
Blank email composer for pitching the dream client or investor.
Character constraints visible on screen. Character responds realistically.
Send tracker shows: opened after 4 hours, no reply. Does learner follow up?

**L. The One-on-One (All tracks — as curveball at T+18)**
Manager: "Hey, do you have 5 minutes?" Simulated 1:1. Manager gives indirect feedback.
Learner must navigate: be honest without seeming incompetent, ask for help without
seeming helpless. Tests self-awareness and managing up simultaneously.

## SECTION 4: THE 8-DIMENSION DIAGNOSTIC SYSTEM

### The 8 Dimensions (Universal Across All Tracks)

```
1. COMMUNICATION CLARITY
Say the right thing to the right person at the right level of detail.
Technical-to-business translation. Appropriate register. No jargon with non-tech people.

2. ESCALATION JUDGMENT
Know when to solve it yourself, loop in your manager, or alert the team.
Most consequential invisible skill. Almost never taught.

3. PRIORITIZATION UNDER PRESSURE
Work on the most important thing. Identify P0 vs P2 correctly.
Deprioritize gracefully when required. Don't spend 40 min on styling during an outage.

4. STAKEHOLDER EMPATHY
Understand what people NEED, not just what they SAY.
Detect the panic behind Priya's friendliness. Hear Kiran's anxiety between the lines.

5. PROACTIVE COMMUNICATION
Update people before they ask. Set expectations early.
Tell your manager you're blocked before they wonder why nothing has moved.

6. HONEST ESTIMATION AND INTEGRITY
Say "I'm not sure yet" when you're not sure.
Don't commit to timelines you can't meet to avoid discomfort.
Surface mistakes immediately, don't hide them.

7. CONSTRUCTIVE PUSHBACK
Disagree when you should, respectfully, with evidence.
Don't capitulate when you shouldn't. Don't dig in when you should let go.

8. DOCUMENTATION AND ACCOUNTABILITY
Create a paper trail others can pick up.
Write post-mortems that surface systemic issues, not just solutions.
Log your investigation steps, not just your conclusions.
```

6 buttons: English | हिन्दी | தமிழ் | ಕನ್ನಡ | తెలుగు | मराठी

Note: *"The simulation runs in English — that's part of the training.

### Step 5 — The Cinematic Briefing

After onboarding, a briefing screen that builds real anticipation:

```
  "Good morning, [Name]."

  "It's Day 3 at Kiran Technologies, Bengaluru."
  "You joined as a Junior Software Engineer on Monday."
  "Your inbox has 2 unread messages."
  "One of them is from your manager."
  "One of them is from the client."
  "The production alert fired 8 minutes ago."

  —

  Your team is ready. Are you?

       [ ENTER THE OFFICE ]
```

Each line fades in sequentially. The last line pauses for 1 second before the button
appears. This creates psychological investment. The learner clicks in with their
heart rate slightly elevated. That is exactly the right state to learn in.

{{MOMENTS_LIST}}

- React to emotional tone, not just content
- If learner is evasive: probe. If learner is clear and specific: acknowledge and advance.
- DO NOT break character. You are a real colleague.
- DO NOT give hints. You are not their coach.
- DO NOT positively reinforce low-quality responses just to be encouraging.
  Real colleagues don't do that.

CURRENT SIMULATION STATE: {{SIMULATION_STATE}}
ELAPSED TIME: {{ELAPSED_MINUTES}} minutes into the sim

## SECTION 8: API STRUCTURE

```
POST /api/onboarding/complete
  body: { name, track, background, experience, selfRated, language }
  returns: { sessionId, simulationConfig, personalizationInsights }

POST /api/sim/start
  body: { sessionId }
  returns: { initialMessages[], taskBoard, inboxEmails, briefingText }

POST /api/sim/message
  body: { sessionId, to, content, medium: 'chat'|'email'|'call' }
  returns: { responses[], updatedState, invisibleSkillMomentTriggered? }

POST /api/sim/curveball/trigger
  body: { sessionId, eventType }
  returns: { characterMessage, systemAlert? }

POST /api/sim/artifact/submit
  body: { sessionId, type: 'standup'|'incident_report'|'email'|'document', content }
  returns: { characterReaction, diagnosticNote }

POST /api/sim/end
  body: { sessionId }
  returns: { diagnostic: DiagnosticReport, transcriptAnnotated }

GET  /api/sim/state/:sessionId
     returns: { simulationState }

POST /api/share/generate
  body: { sessionId, platform }
  returns: { shareText, imageUrl }
```

## SECTION 11: RISKS AND MITIGATIONS

**Risk: 8 tracks is too much for a hackathon**
Build SWE + Data Science completely. Show all 8 on the track selection screen
(6 in soft gray as "coming soon"). The infrastructure carries the argument.

**Risk: Personalization feels like a buzzword**
Call it "calibrated to you" in the demo. The mechanism is invisible.
If asked: "We run a Claude API call with the learner's self-assessment profile
to adjust character intensity, curveball timing, and diagnostic dimension weights
before the simulation begins."

**Risk: Demo breaks live**
Cache and pre-generate the first 3 character exchanges (scripted anyway).
Simulation goes "live" after the learner's second message. Have recorded video backup.
Never demo without a backup.

**Risk: "This is just ChatGPT with roles"**
Three-part answer: (1) Characters have hidden agendas that react to learner judgment.
(2) Curveball events fire on timers independent of learner messages.
(3) The diagnostic gives 8-dimension gap taxonomy with specific transcript citations.
Then show the diagnostic card. That ends the objection.
