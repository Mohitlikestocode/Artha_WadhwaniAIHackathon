**Service Worker Strategy (sw.js) — Three-Layer Caching:**

```javascript
// LAYER 1: App Shell (install-time cache) — always offline available
const APP_SHELL_CACHE = 'karmaloop-shell-v1';
const APP_SHELL_FILES = [
  '/',
  '/index.html',
  '/assets/index.js',       // Vite bundle
  '/assets/index.css',
  '/fonts/inter.woff2',
  '/fonts/jetbrains-mono.woff2',
  '/icons/icon-192.png',
  '/offline.html',           // Beautiful offline page, not browser default
  // All 8 track config JSONs pre-cached:
  '/data/tracks/swe.json',
  '/data/tracks/cybersecurity.json',
  '/data/tracks/data-science.json',
  '/data/tracks/marketing.json',
  '/data/tracks/ux-design.json',
  '/data/tracks/pentesting.json',
  '/data/tracks/freelancer.json',
  '/data/tracks/founder.json',
  // All Branch Engine scenarios for offline play:
  '/data/scenarios/swe-branch.json',
  '/data/scenarios/freelancer-branch.json',
  // The full jargon library:
  '/data/jargon-library.json',
  // Inbox Zero email sets (pre-generated, no API needed):
  '/data/inbox/swe-monday.json',
  '/data/inbox/marketing-monday.json',
];

// LAYER 2: Runtime cache — API responses cached as they arrive
const RUNTIME_CACHE = 'karmaloop-runtime-v1';

// LAYER 3: Simulation queue — actions taken offline, synced when back online
const SYNC_QUEUE = 'karmaloop-sync-queue';

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(APP_SHELL_CACHE)
      .then(cache => cache.addAll(APP_SHELL_FILES))
      .then(() => self.skipWaiting())
  );
});

self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // Claude API calls — cannot work offline, must queue
  if (url.hostname === 'api.anthropic.com') {
    event.respondWith(handleAIRequest(request));
    return;
  }

  // App shell — serve from cache first
  if (APP_SHELL_FILES.includes(url.pathname)) {
    event.respondWith(
      caches.match(request).then(cached => cached || fetch(request))
    );
    return;
  }

  // Everything else — network first, fall back to cache
  event.respondWith(
    fetch(request)
      .then(response => {
        const clone = response.clone();
        caches.open(RUNTIME_CACHE).then(cache => cache.put(request, clone));
        return response;
      })
      .catch(() => caches.match(request) || caches.match('/offline.html'))
  );
});

async function handleAIRequest(request) {
  try {
    // Try network first
    const response = await fetch(request.clone());
    return response;
  } catch {
    // Offline: queue the action for sync later
    const body = await request.clone().json();
    await addToSyncQueue(body);
    
    // Return an offline response using pre-generated content
    return new Response(
      JSON.stringify(getOfflineFallbackResponse(body)),
      { headers: { 'Content-Type': 'application/json' } }
    );
  }
}

// Background sync when connection restores
self.addEventListener('sync', (event) => {
  if (event.tag === 'karmaloop-sync') {
    event.waitUntil(processSyncQueue());
  }
});
```

## PART 2: ADAPTIVE COMPREHENSION THRESHOLD SYSTEM

### The Problem

Three students open KarmaLoop:
- **Ananya** from IIT Delhi, did 2 internships, used Jira before. Needs intensity.
- **Ravi** from a Tier 3 engineering college in MP, first time hearing "standup." Needs scaffolding.
- **Fatima**, a BCA dropout who taught herself Python on YouTube. Somewhere between them.

A single simulation difficulty setting doesn't serve all three. The adaptive system
must detect — in real time, during the simulation — where each learner is struggling
and adjust accordingly. Not just based on their self-assessment at onboarding.

### Layer 1: Pre-Simulation Calibration (already in V2, now extended)

The onboarding self-assessment gives us a starting estimate. But self-assessments
are unreliable. Indian students systematically underestimate themselves (trained
humility) or overestimate (trained to project confidence). We can't trust these numbers alone.

**The Calibration Round (NEW in V4):**
Before the full simulation, the learner does a 3-minute "calibration sprint."
5 short situations. Each has a clear professional response and a clear amateur response.
Based on their answers, we recalibrate their starting difficulty.

```javascript
// Calibration Sprint scenarios — quick-fire, 30 seconds each
const CALIBRATION_SCENARIOS = [
  {
    prompt: "Your manager asks: 'Can you send me the status update by 3pm?' It's 2:48pm and you won't have it ready. What do you do?",
    options: [
      { text: "Quickly send whatever you have at 3pm", score: 2 },
      { text: "Message them now: 'I need until 3:30pm. Is that okay?'", score: 5 },
      { text: "Send it at 3:15pm without warning", score: 1 },
      { text: "Ask a colleague to send it for you", score: 0 }
    ]
  },
  // ... 4 more
];

function calculateStartingDifficulty(calibrationScores) {
  const avg = calibrationScores.reduce((a, b) => a + b, 0) / calibrationScores.length;
  if (avg >= 4) return 'intense';
  if (avg >= 2.5) return 'standard';
  return 'gentle';
}
```

### Layer 2: Real-Time Comprehension Detection (NEW in V4)

During the simulation, the system monitors signals that indicate the learner is
struggling or thriving. Every 3 minutes, a silent "comprehension check" runs.

**Struggle signals (triggers difficulty decrease):**
- Response time > 90 seconds on a simple message (they're frozen)
- Learner uses the word "sorry" 3+ times (over-apologizing = anxiety)
- Learner's messages get shorter over time (withdrawal from overwhelm)
- Learner ignores 2+ messages from characters (paralysis)
- Learner asks characters for help on every decision (dependency)
- Jargon tooltips accessed 8+ times in first 5 minutes (vocabulary overload)

**Thriving signals (triggers difficulty increase):**
- Response time < 20 seconds consistently (too easy)
- All characters responding positively (not enough challenge)
- Zero jargon lookups (vocabulary comfortable)
- Learner proactively communicates without being asked (ahead of curve)
- Learner correctly identifies and handles all invisible skill moments

**The Adaptation Response (invisible to learner):**

```javascript
// ComprehensionAdaptor.js — runs silently every 3 minutes
async function adaptDifficulty(sessionState) {
  const signals = analyzeSignals(sessionState);
  const currentDifficulty = sessionState.difficulty;
  
  if (signals.struggleScore > 0.7 && currentDifficulty !== 'gentle') {
    // Dial down silently
    await updateCharacterInstructions(sessionState, {
      tone: 'warmer',
      messageFrequency: 'slower',
      curveballDelay: '+5 minutes',
      helpHints: 'enabled'
    });
    
    // Inject a "scaffolding moment" — a character naturally explains context
    // Example: Rajan says "Just so you know as a new joiner — when production
    //   alerts fire, the first thing we always do is check the logs before
    //   touching anything else."
    await injectScaffoldingMoment(sessionState);
    
    sessionState.adaptationLog.push({
      time: sessionState.elapsed,
      action: 'difficulty_decreased',
      reason: signals.primaryStruggleSignal
    });
  }
  
  if (signals.thrivingScore > 0.8 && currentDifficulty !== 'intense') {
    // Dial up silently
    await updateCharacterInstructions(sessionState, {
      tone: 'more_demanding',
      curveballDelay: '-3 minutes',
      additionalCurveball: 'enable_layer2_curveball'
    });
    
    sessionState.adaptationLog.push({
      time: sessionState.elapsed,
      action: 'difficulty_increased',
      reason: 'learner_performing_well'
    });
  }
}
```

### Layer 3: The Scaffolding Moments (NEW in V4)

When the system detects a struggling learner, it doesn't lower the simulation's
emotional reality — it injects **natural, in-character scaffolding** through the
senior character (Rajan, Divya, Anjali — whoever is the wise mentor in the track).

These scaffolding moments feel like good mentorship, not like the game being made easier.

**Examples of scaffolding moments by track:**

SWE Track:
```
Rajan [in DM, after detecting learner is frozen on the P0]:
"Hey — not to put pressure but just a tip from my first week here: when you're
stuck on a prod issue, the fastest thing is always to check the logs first before
anything else. Once you see the error, you know where to look. Does that help?"
```

Data Science Track:
```
Ritu [your manager, after detecting learner hasn't responded to Meera's analysis request]:
"Quick tip before you dive in — Meera tends to have a narrative in her head before
she sees data. Your job isn't to confirm her story, it's to find out what the data
actually says and then bring her along. Does that make sense?"
```

Freelancer Track:
```
Your friend / advisor character [injected when negotiation stalls]:
"Hey — just thinking about you. When a client comes in low on budget, the worst
thing you can do is immediately say no or immediately say yes. The best move is
almost always to ask what they need delivered and then see if you can reframe the value."
```

### Layer 4: The Comprehension Check (Between Modes)

Between simulation modes (e.g., after The Office, before War Room), the system
shows a 60-second "reflection check" — 3 micro-questions about what just happened:

```
Before you move to the next mode, quick check:

1. In your simulation, Priya asked for an ETA when you didn't have one yet.
   What was the most professional way to handle that?
   
   ○ Give your best guess
   ● Ask for time and give a specific callback: "Let me check the logs. 
     I'll update you by 10:15."
   ○ Say "I'm working on it"
   ○ Don't respond until you have an answer
   
   [Why this answer matters: brief 2-sentence explanation]

2. Kiran admitted he pushed to main by accident. What should you have done?
   ...

3. The production alert fired at the worst possible time. What's always 
   the first step when prod breaks?
   ...
```

These aren't tests. They're **retrieval practice** — the most effective learning
technique known to cognitive science. Forcing recall of what just happened deepens
retention by 40-60% compared to passive review. The answers are shown immediately
with a brief explanation. No scores, no grades — just "here's why."

The Comprehension Check also updates the learner's profile: if they get all 3 right,
the next mode increases in difficulty. If they get 1 or fewer right, the next mode
includes extra scaffolding moments.

### Layer 5: Spaced Repetition for Skills (NEW in V4)

The platform tracks which invisible skills the learner struggles with across sessions.
Using an SM-2-inspired spaced repetition algorithm:

```javascript
// SpacedRepetitionEngine.js
const SM2 = {
  calculateNextReview(dimension, currentScore, previousInterval, easeFactor) {
    // If score < 3: repeat in 1 day (failed recall)
    // If score >= 3: use SM-2 formula for next interval
    if (currentScore < 3) {
      return { nextInterval: 1, newEaseFactor: Math.max(1.3, easeFactor - 0.2) };
    }
    
    const newInterval = previousInterval === 0 ? 1 :
                        previousInterval === 1 ? 6 :
                        Math.round(previousInterval * easeFactor);
    
    const newEaseFactor = easeFactor + (0.1 - (5 - currentScore) * (0.08 + (5 - currentScore) * 0.02));
    
    return {
      nextInterval: newInterval,
      newEaseFactor: Math.max(1.3, newEaseFactor),
      nextReviewDate: addDays(new Date(), newInterval)
    };
  }
};

// The system uses this to:
// 1. Schedule "return visits" to dimensions the learner scores low on
// 2. Send push notifications: "Time to practice Escalation Judgment again"
// 3. Adjust which scenarios are served next (serve scenarios that exercise weak dimensions)
```

**Push notification example (PWA push via service worker):**
```

Your last score: Emerging. 5-minute drill ready. 🎯"
```

### The Problem

The diagnostic exists only in the learner's app. It has zero value to an employer
who hasn't heard of KarmaLoop. The gap between "credentials earned" and "capabilities
demonstrated" (the exact words of Problem 3.2) won't close until employers can
SEE the capability evidence.

### The KarmaLoop Credential (Verifiable, Shareable, Employer-Readable)

After completing at least 3 simulation modes on any track, the learner earns a
**KarmaLoop Capability Certificate**. This is not a badge. It is a structured,
verifiable document that contains:

1. **The learner's name and track**
2. **The 8-dimension radar chart** (as a visual)
3. **Their top 3 strongest dimensions** with specific evidence quotes from their simulation
4. **Their growth trajectory** (if they've done multiple sessions, shows improvement)
5. **The simulated employer signal** — one-sentence hiring manager assessment
6. **A unique verification URL**: karmaloop.in/verify/[UUID]

The verification URL shows any employer who visits: the learner's capability profile,
the date of their last simulation, how many modes they've completed, and whether their
scores are improving over time.

## PART 4: REAL-TIME COACHING (THE INNER VOICE)

### The Problem

V1–V3 evaluated the learner after the simulation. But the most powerful learning
happens IN THE MOMENT — when you're about to make a mistake, not after you've made it.

The challenge: if we interrupt the simulation to coach the learner, we break immersion.
If we don't, they practice bad habits without correction.

**The solution: The Inner Voice — a non-intrusive, opt-in real-time coach.**

### How The Inner Voice Works

A small, minimizable panel sits at the bottom-left of the simulation room.
It appears as a subtle glow (Indigo border, dark background), not a bright notification.

**The Inner Voice is NOT:**
- An AI assistant answering questions
- A hint system that makes the simulation easier
- A pop-up interruption

**The Inner Voice IS:**
- A real-time reflection prompt that appears 5 seconds after the learner types but
  BEFORE they send their message
- It reflects back what they're about to say and asks ONE question
- After they send, it reflects on what happened

**Example (learner has typed "sure, I'll look into it" to Priya's ETA request):**

```
┌─────────────────────────────────────────────────┐
│  💭 Inner Voice                          [hide] │
│                                                 │
│  You're about to say: "Sure, I'll look into it" │
│                                                 │
│  Quick thought: Does this give Priya a          │
│  specific ETA she can tell the client?          │
│                                                 │
│  [Send anyway]    [Revise my message]           │
└─────────────────────────────────────────────────┘
```

The learner can: ignore it and send, click "Send anyway" (the voice logs that
they saw the prompt but proceeded), or click "Revise" (the text input stays open).

**The Inner Voice does NOT tell the learner what to say.** It asks a question.
The learner must still figure out the answer.

**Post-send reflection (appears 3 seconds after message sent):**
```
┌─────────────────────────────────────────────────┐
│  💭 Inner Voice                          [hide] │
│                                                 │
│  Priya replied in 8 seconds with another        │
│  question. That usually means she didn't        │
│  get the answer she needed.                     │
│                                                 │
│  What might you have said differently?          │
│  [You'll see in your diagnostic →]              │
└─────────────────────────────────────────────────┘
```

**The Inner Voice is disabled in "Assessment Mode"** — for learners who want a
pure evaluation without coaching. Assessment Mode is chosen at mode start:

```
"How do you want to approach this simulation?"
  ○ Learning Mode (Inner Voice enabled — coaching as you go)
  ○ Assessment Mode (No hints — pure evaluation)
```

Learning Mode is recommended for first-time users.
Assessment Mode is recommended for repeat sessions to test genuine growth.

### Inner Voice Trigger Logic

```javascript
// InnerVoiceEngine.js
async function analyzeBeforeSend(draftMessage, simulationState) {
  // Only trigger if draft shows a potential invisible skill gap
  const triggers = [
    {
      pattern: /sure|ok|no problem|will do|on it/i,
      context: 'response_to_eta_request',
      question: 'Does this give your stakeholder a specific time they can count on?'
    },
    {
      pattern: /sorry|my bad|apologize/i,
      context: 'any',
      question: 'Is an apology what this situation needs, or is an action plan more useful here?'
    },
    {
      pattern: /^.{1,15}$/,  // Very short response (< 15 chars) to a complex message
      context: 'response_to_complex_situation',
      question: 'This situation has a few moving parts. Does your reply address the most important one?'
    },
    {
      pattern: /i think|maybe|i guess|probably/i,
      context: 'response_to_technical_question',
      question: 'Confidence matters here. If you\'re not certain, could you give a specific time when you will be?'
    }
  ];
  
  for (const trigger of triggers) {
    if (trigger.pattern.test(draftMessage) && matchesContext(trigger.context, simulationState)) {
      return { shouldTrigger: true, question: trigger.question };
    }
  }
  
  return { shouldTrigger: false };
}
```
